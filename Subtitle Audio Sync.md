## 🔍 1. สาเหตุหลักที่เกิดขึ้น (Root Cause Analysis)

จากที่คุณแจ้งอาการ: **"ซับไตเติลค่อยๆ เหลื่อมจากเสียง (ช่วงแรกตรง แต่ท้ายคลิปเริ่มไม่ตรง)"**
เมื่อเราเข้าไปวิเคราะห์ไฟล์ `gemini.js`, `audio-sync.js` และ `app.js` พบสาเหตุหลักดังนี้ครับ:

### ❌ ปัญหาที่ 1: AI Audio Sync ล้มเหลวตลอดเวลา (Silent Failure)
ในฟังก์ชัน `geminiSrtFromAudio` (ไฟล์ `gemini.js`) โค้ดเดิมส่งข้อมูลเสียง WAV Base64 ไปยัง OpenRouter API ด้วยโครงสร้างแบบรูปภาพ:
```json
{
  "type": "image_url",
  "image_url": {
    "url": "data:audio/wav;base64,..."
  }
}
```
* **ผลกระทบ**: โครงสร้างแบบนี้ผิดมาตรฐานของ OpenAI/OpenRouter สำหรับข้อมูลเสียง ส่งผลให้โมเดล Gemini ไม่ได้รับไฟล์เสียงจริง และทำให้การเรียก AI-driven Sync ล้มเหลว (เกิด Error และลงประวัติในระบบ Debug Logs เสมอ)
* **ผลลัพธ์**: ระบบต้องตกไปทำงานที่ **Deterministic Fallback Sync** (ในไฟล์ `audio-sync.js`) แทน

### ❌ ปัญหาที่ 2: การกระจายแบบเส้นตรง (Linear Distribution Fallback)
เมื่อ AI Sync ล้มเหลว ฟังก์ชัน fallback ใน `audio-sync.js` (`buildContinuousTextSegments`) จะทำหน้าที่คำนวณแบ่งเวลาตามสัดส่วนความยาวของตัวอักษร (Text Weight) แบบเส้นตรง:
* **ผลกระทบ**: เนื่องจากเสียงพากย์ไทยจริงไม่ได้พูดด้วยความเร็วคงที่สม่ำเสมอตลอดทั้งประโยค มีจังหวะวรรคและเน้นเสียง ทำให้การกระจายเวลาแบบเฉลี่ยอักษรค่อยๆ สะสมความคลาดเคลื่อน ยิ่งช่วงท้ายคลิปซับจะยิ่งเหลื่อมช้าหรือเร็วกว่าเสียงจริงอย่างเห็นได้ชัด

## 🛠️ 2. สิ่งที่ได้รับการแก้ไขแล้ว (Implemented Fixes)

เราได้ดำเนินการแก้ไขในไฟล์ [gemini.js](file:///c:/Users/usEr/OneDrive/Desktop/work/Do-Subtitle/do-subtitle-webapp/gemini.js) เรียบร้อยแล้ว โดยปรับโครงสร้างการส่งเสียงให้ถูกต้องตามมาตรฐานมัลติโมเดลของ OpenRouter/Gemini:

```javascript
// ปรับปรุงการส่ง payload ใน gemini.js
const payload = {
  model: model,
  messages: [
    {
      role: "user",
      content: [
        {
          "type": "text",
          "text": promptText
        },
        {
          "type": "input_audio",      // แก้ไขเป็นประเภท input_audio ที่ถูกต้อง
          "input_audio": {
            "data": base64Wav,        // ส่ง Base64 เปล่าๆ
            "format": "wav"           // ระบุฟอร์แมต wav
          }
        }
      ]
    }
  ]
};
```
* **ผลลัพธ์หลังแก้ไข**: 
  1. AI Gemini 2.5 Flash จะสามารถได้ยินและฟังคลื่นเสียงพากย์จริงได้อย่างชัดเจนร้อยเปอร์เซ็นต์
  2. ตัว AI จะระบุตำแหน่งเวลาเริ่มต้นและสิ้นสุดของแต่ละคำพากย์ (Timecode) ออกมาเป็นไฟล์ SRT ส่งกลับมา ทำให้ซับไตเติลเต้นตรงตามจังหวะคำพากย์จริงอย่างแม่นยำ ไม่เจอกับความเหลื่อมสะสมตอนท้ายคลิปอีกต่อไป

## 💡 3. คำแนะนำในการพัฒนาเพิ่มเติมเพื่อความแม่นยำสูงสุด (Future Recommendations)

หากต้องการให้แอปพลิเคชันเสถียรและทนทานต่อเคสเสียงเหลื่อมมากยิ่งขึ้น แนะนำให้ปรับปรุงเพิ่มเติมในจุดต่างๆ ดังนี้:

### A. เพิ่มช่องกรอกปรับเลื่อนเวลาซับไตเติล (Subtitle Time Offset Shift)
บางครั้ง ตัวเบราว์เซอร์หรือโมดูลการเล่นวิดีโออาจมีอาการดีเลย์คงที่ (Constant Latency) เช่น ซับช้ากว่าเสียง 0.3 วินาทีเท่ากันทุกบรรทัด
* **แนวทางแก้ไข**: เพิ่ม input หรือปุ่มกดปรับแต่ง `Time Offset` (เช่น `+0.1s`, `-0.1s`) ในการตั้งค่าหน้าเว็บ เพื่อให้ผู้ใช้งานสามารถเลื่อนซับทั้งหมดเดินหน้า/ถอยหลังพร้อมกันในครั้งเดียวตอนฝังวิดีโอ

### B. ควบคุมความสั้น-กระชับของประโยคพากย์ (Strict Length Guidelines)
ในกรณีที่ AI Sync พลาดและต้องหล่นลงไปใช้ Fallback ของระบบจริง:
* **แนวทางแก้ไข**: ควรปรับ Prompt ในขั้นวิเคราะห์ภาพเขียนบท (Step 3) ให้พยายามแบ่งคำบรรยายแยกเป็นท่อนสั้นๆ ไม่เกิน 15-18 ตัวอักษรต่อบรรทัด ยิ่งประโยคสั้น การเกลี่ยเฉลี่ยเวลาแบบเส้นตรงจะคลาดเคลื่อนน้อยลงและดูตรงจังหวะปากมากขึ้น
